Architecture Overview

Four abstraction layers, each behind a trait for testability. Commands enter via main.rs, dispatched through the CLI layer, resolved by core logic that delegates I/O to git and tmux abstractions.

Config Resolution

Settings are resolved across 5 layers at startup by core/config.rs::resolve_config(). Higher layers shadow lower ones. The result is an EffectiveConfig passed to every handler.

Worktree Lifecycle

A worktree is a coordinated set of git worktrees across all packages sharing a branch name, tracked in .meldr/state.json.

Full Lifecycle

stateDiagram-v2 [*] --> Empty : meldr init Empty --> HasPackages : meldr package add url git clone --bare to packages/pkg/ HasPackages --> WorktreeActive : meldr worktree add branch git worktree add + tmux window WorktreeActive --> WorktreeActive : meldr sync fetch + rebase, snapshot saved WorktreeActive --> WorktreeActive : meldr exec cmd run in each pkg worktree dir WorktreeActive --> HasPackages : meldr worktree remove branch clean check + prune + kill tmux HasPackages --> HasPackages : meldr package remove pkg
